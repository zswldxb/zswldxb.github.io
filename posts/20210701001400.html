Memory-Efficient Implementation of DenseNets

Preface

Paper

Purpose

本文指出: 由于特征重用, DenseNet[1] 网络架构的计算效率很高, 但是, 如果管理不当, 可能会产生大量的空间浪费. 由于特征图源自于每层生成的中间特征图, 为 batch normalization 和 concatenation 操作的输出, 在前向传播和反向传播过程中都需要使用中间特征图. 如果没有妥善的管理, 其所占内存为层数的二次函数.

本文主要目的为减少 DenseNet 模型在训练期间的内存开销.

The DenseNet Architecture

DenseNet 由 dense block, transition layer 和 classification layer 组成, 其中 dense block 中的每一层都接收所有前面的层的特征图作为输入. 即对于第 lll 层:

xl=Hl([x0,⋯ ,xl−1])x_{l}=H_{l}([x_{0},\cdots,x_{l-1}])xl​=Hl​([x0​,⋯,xl−1​])

其中 HlH_{l}Hl​ 表示第 lll 层的操作, 而 [⋅][\cdot][⋅] 表示 concatenation 操作. 在 DenseNet 定义 HlH_{l}Hl​ batch normalization, recitified linear unit(ReLU) 和 convolution 操作的组合. 在大多数深度学习框架中, 这三个操作中的每一个都产生 intermediate feature map. 此外 [⋅][\cdot][⋅] 也会差生 intermediate feature map. 很多深度学习框架将这些中间特征图保存在 CPU 内存中, 以便反向传播时使用.

Pre-activation batch normalization

在 DenseNet 中, 采用 pre-activation batch normalization, 即在卷积运算之前使用 batch normalization 和 ReLU, 这对 DenseNet模型效果的提升是重要的, 然而 pre-activation batch normalization 会产生中间特征图.

Contiguous concatenation

当卷积运算的输入位于连续的内存块中时, 其运算是高效的. 有的深度学习框架(如 Torch)明确要求卷积操作的输入必须为连续内存块, 有的(如 CUDNN)则没有这要求, 但是使用非连续内存块会增加 30%-50% 的开销.

为了使用连续的输入, 每层必须将以前的特征复制到连续的内存块中, 对于具有 mmm 层的网络, 每个特征最多复制 mmm 次, 如果单独存储, 则 concatenation 操作会产生 O(m2)O(m^{2})O(m2) 的内存开销.

值得注意的是, 不能简单的讲卷积的输出分配给连续的内存块. 对于表示为 Rn×d×w×h\R^{n\times d\times w\times h}Rn×d×w×h 或 Rn×w×h×d\R^{n\times w\times h\times d}Rn×w×h×d, 其中 nnn 为 mini-batch 的样本数, ddd 为 feature map 的数量, www 和 hhh 为 feature map 的宽度和高度. concatenation 操作应该沿着特征维度而非样本维度进行拼接.

Naive Implementation

对于 Naive Implementation, 其将为每一个操作的结果分配一个连续的内存空间. 对于上图左侧的 concatenation 操作, 其将复制输入特征图, 并按特征维度拼接后存储到一块连续的内存块中, 如果第 lll 层前面的每一层生成 kkk 个特征图, 则连续的内存块必须能容纳 l×kl\times kl×k 个特征图, 然后将这些特征图输入到 batch normalization 操作. 此时同样需要分配一块能存储 l×kl\times kl×k 个特征图的连续的内存块, 以存储 batch normalization 操作的输出(ReLU 为 in-place 操作, 因此不考虑). 最后经过卷积操作输出 kkk 个新的特征. 可以看出, 对于第 lll 层操作, 中间特征图需要 O(l×k)O(l\times k)O(l×k) 的内存, 而输出特征只需要恒定的 O(k)O(k)O(k) 的内存.

此外, 在反向传播过程中也会产生大量的中间梯度, 存储这些梯度, 需要 O(l×m)O(l\times m)O(l×m) 的内存.

Memory-Efficient Implementation

由于 batch normalization 和 concatenation 操作的计算时间远快于卷积操作, 因此, 本文提出使用两个预先分配的 Shared Memory Storage 来避免内存的二次增长. 在前向传播期间, 我们将所有的中间输出分配到这些内存块中, 在反向传播期间, 我们将根据需要动态重新计算 batch normalization 和 concatenation 的特征图.

这一策略对于 DenseNet 非常有效. batch normalization 和 concatenation 操作需要使用大量的内存, 然而其计算时间仅为总计算时间的一小部分. 因此, 重新计算这些中间输出可以节省大量内存, 而且计算时间开销很小.

参考文献
